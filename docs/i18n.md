# Internationalization (i18n)

Libraries & Structure
- Library: `next-international` with server/client helpers.
- Config: `src/i18n/config.ts:1` defines `locales`, `defaultLocale`, and `rtl`.
- Helpers: `src/i18n/next-international.server.ts:1`, `src/i18n/next-international.client.ts:1`.
- Messages: `messages/<locale>/<namespace>.json` (e.g., `messages/en/common.json:1`).

Locale Routing
- Locale is the first segment: `src/app/[locale]/**`.
- Middleware negotiates locale from header/cookie/path: `middleware.ts:1`.
- Generate static params for locales using `getStaticParams` from the i18n server helper.

Using translations
- Server: `const t = await getI18n()` → `t('key')`.
- Client: `const { useI18n } = createI18nClient(...)` → `const t = useI18n()`.
- Scoped keys: `getScopedI18n('home')` or `useScopedI18n('home')`.

Adding a new locale
1. Add the locale to `NextIntlConfig.locales` and `rtl` if applicable (`src/i18n/config.ts:1`).
2. Create message files: `messages/<new-locale>/{common,home,index}.json`.
3. Run `pnpm i18n:check` to ensure key parity.
4. Ensure assets/metadata handle the locale (SEO, `src/lib/seo.ts:1`).

Revalidate localized content
- Endpoint: `POST /api/site/revalidate` revalidates localized pages (`src/app/api/site/revalidate/route.ts:1`).
